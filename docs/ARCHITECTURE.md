# Yomu 架构

Yomu 的原则是“客户端轻量，不等于功能精简”。最终产品保留 Reader 3 的成熟能力，计算和持久化尽可能在自托管服务器完成，手机与浏览器只负责交互、排版和有限离线缓存。

面向当前个人与少量用户的纯阅读场景，产品明确不提供 Kindle 仿真和笔记编辑；书签只同步章节与阅读位置。这个取舍不会削减书源、搜索、换源、阅读排版、缓存和多端同步等阅读主链路。

## 运行分层

1. **PWA 客户端**：同源登录、书架、搜索、阅读、手势、主题、排版、TTS 控制和按用户/站点隔离的 IndexedDB 整本缓存。客户端不保存 Reader 令牌或管理密钥。
2. **Yomu App（Node）**：一个进程同时提供 PWA/SSR、静态压缩和安全网关；把 Reader 令牌封装进 AES-GCM 加密、HttpOnly、Secure、SameSite 会话 Cookie，并执行 CSRF/Origin、登录限速、管理员复核、请求限制和危险参数剥离。
3. **Reader Core（Rust）**：本项目直接维护的服务端源码，负责账号、书架、分组、进度、书签、书源、并发搜索、换源、正文缓存、净化规则、RSS、WebDAV、本地书和权限。关闭公开注册与 AI，并保持 `/reader3` 数据接口。
4. **WebView Sidecar（可选）**：只处理显式声明 `webView`、需要页面脚本、验证码或登录 Cookie 的书源。普通 HTTP/CSS/JSONPath/XPath/Regex/QuickJS 规则永远不拉起浏览器进程。
5. **SQLite + 文件存储**：适合个人和少量用户。SQLite 开 WAL、定期 checkpoint；章节缓存与上传书籍使用文件目录，备份覆盖数据库和文件清单。

## 完整功能矩阵

| 领域 | 能力 | 执行位置 |
| --- | --- | --- |
| 账户同步 | 私有登录、管理员开户、多用户隔离、书架和进度 | Gateway + Server |
| 书源 | 导入导出、编辑、分组、启停、检测、登录、Cookie、调试 | Server + WebView Sidecar |
| 搜索发现 | 单源/多源并发搜索、书海、去重、换源 | Server |
| 阅读器 | 滚动/分页、手势、主题、字体、净化、简繁、书签、章节内查找 | PWA + Server |
| 媒体 | 漫画、音频、TTS、媒体请求代理 | PWA + Server |
| 离线 | 浏览器章节缓存、服务端整本缓存、缓存管理 | PWA + Server |
| 本地书 | TXT、EPUB、MOBI、PDF、导入导出与本地书仓 | Server |
| 内容流 | RSS 源、文章列表、统一阅读页 | Server + PWA |
| 维护 | WebDAV、备份恢复、健康检查与受控日志 | Server |

## 搜索与书架更新的资源边界

- 多书源搜索使用 SSE 边搜边显示，客户端同一时间只保留一个搜索控制器；新搜索、退出或页面中止会取消旧连接。
- 客户端默认请求 80 条、并发 6；Core 对单次请求设并发 8、结果 200 的硬上限，并以全局 16 个出站槽限制多用户叠加压力。HTTP 连接池每域只保留 2 个空闲连接。每个来源返回的数据先按剩余配额截断，再去重发送，连接关闭后停止派发任务。
- 书架首屏只渲染 48 本，按需继续显示，避免大书架一次创建过多 DOM。
- 更新书架只刷新目录元数据，不抓取全部正文；服务端默认并发 3、硬上限 4，跳过本地书和不可更新项，完成后一次保存。客户端用互斥状态阻止重复点击。
- 分类发现默认关闭，不做后台扫描；启用后也只在进入分类页面时按页请求。

## 完全离线

Service Worker 只缓存版本化应用壳和带哈希的静态资源，明确排除登录、API 和用户资源响应。整本下载以两个工作任务逐章写入 IndexedDB；断网冷启动时客户端从当前站点、当前用户名的命名空间恢复书架、目录、正文和进度。离线状态只允许阅读与本机进度更新，不伪装执行书源、用户或服务器数据写入；恢复网络后再使用正常同步链路。

这使安装后的 PWA 足以充当轻量客户端：不常驻后台服务，也不额外打包一套浏览器内核。原文模式不加载简繁词典；需要转换时才按方向加载对应词典，其中繁转简使用明显更小的独立数据块。

## WebView 策略

WebView 是兼容层，不是默认网络栈。服务端规则解析先走轻量执行器；只有规则明确要求 `webView`，或登录流程需要真实浏览器上下文时，才向 Sidecar 发任务。Sidecar 应设并发上限、空闲自动退出、每用户/书源隔离 Cookie、域名白名单和任务超时。手机 App 将来可用系统 WebView 完成一次登录并把 Cookie 加密同步到服务器，但日常阅读仍由 PWA 完成。

实现位于 `services/webview-bridge` 与 `services/reader-core`。旁车把桥接服务和 Chromium 放在同一个受限容器内，不开放浏览器调试口；它会拦截私网、回环和云元数据目标，限制并发与等待时长，并默认阻止图片、媒体和字体。

## 小规模自托管默认值

- 1–10 个用户：单实例 Rust Core + SQLite WAL 足够，默认不引入 PostgreSQL、Redis 或消息队列。
- WebView Sidecar 默认关闭；启用时建议服务器至少 2 GB 内存，并限制为 1 个并发页面。
- 默认只常驻 Yomu App 与 Reader Core 两个容器；App 负责同源路由和安全头，Core 不开启 CORS，也不直接暴露端口。保留 Core 容器边界，是为了隔离会执行第三方书源规则和发起外部请求的高风险工作负载。
- 所有持久数据位于 `reader-storage` volume 的 `/app/storage`；备份策略由部署者按数据价值和服务器条件配置。
